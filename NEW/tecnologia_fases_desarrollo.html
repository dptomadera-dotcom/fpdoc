Stack recomendado — criterio: máxima productividad con equipo pequeño

Orientado a un desarrollo ágil por un equipo de 1–3 personas, con tecnologías maduras, documentación amplia y despliegue sencillo en infraestructura educativa.

Frontend

Next.js 15 (React)
Framework fullstack con SSR. Permite construir la UI y las API routes en el mismo proyecto. Ideal para aplicaciones con roles y dashboards complejos.
Tailwind CSS
Sistema de diseño basado en utilidades. Permite implementar el sistema visual de FPdoc con coherencia y rapidez sin CSS personalizado extenso.
shadcn/ui
Componentes accesibles y personalizables (tablas, formularios, modales). No es una librería de dependencia — el código vive en el proyecto.
Recharts
Gráficos de cobertura transversal, barras de progreso y mapas de calor del mapa de transversalidad. Ligero y compatible con React.

Backend y base de datos

Supabase (PostgreSQL)
Base de datos relacional con autenticación, Row Level Security por rol y API REST generada automáticamente. Perfecto para el modelo de datos de FPdoc con sus relaciones complejas entre entidades.
Prisma ORM
Define el esquema de base de datos en código TypeScript. Genera migraciones automáticas y tipos seguros para todas las entidades (MODULO, RA, CRITERIO, UT…).
NextAuth.js
Gestión de sesiones y autenticación por rol (JEFE, DOCENTE, ALUMNO). Compatible con SSO institucional (Google Workspace, Microsoft Entra) habitual en centros educativos.
Resend (email)
Envío de notificaciones del sistema de estados (borrador → revisión, cambios solicitados, aprobación). API simple, plantillas React compatibles.

Generación de documentos y despliegue

React PDF / Puppeteer
Generación del PDF de la Programación Didáctica completa (17 secciones). React PDF para plantillas en componentes React; Puppeteer como alternativa para fidelidad exacta al diseño.
pdf-parse / pdfjs
Extracción de RA y criterios desde los PDFs de los módulos que ya tiene el profesorado (ATZ_RA.pdf, PMB_RA.pdf…). Reduce el trabajo de introducción manual.
Vercel
Despliegue con un clic desde GitHub. Previews automáticos por rama para testing. Plan gratuito suficiente para piloto; escalable sin reconfiguración.
TypeScript
Tipado estático en todo el proyecto. Especialmente crítico aquí por la complejidad del modelo de datos (RA, criterios, UT, cambios) y el sistema de permisos por rol.

Cada fase entrega valor real y puede usarse de forma independiente. El objetivo es tener un piloto funcional con un departamento real al final de la Fase 2.

Riesgos identificados y decisiones clave

Los riesgos más relevantes no son técnicos — son de adopción. La tecnología más sofisticada falla si el profesorado no la usa. El diseño del MVP está pensado para minimizar esa fricción.

Riesgos técnicos

⚡
Extracción de RA desde PDF poco fiable
Los PDFs de los módulos no tienen estructura estandarizada. La extracción automática será imperfecta — hay que diseñar un flujo de revisión manual rápida, no depender del 100% de acierto.
⚡
Generación de PDF con fidelidad al formato oficial
Las programaciones tienen formatos específicos que varían por comunidad autónoma. El PDF generado debe ser aceptable ante inspección — requiere iteración con docentes reales durante el piloto.

Riesgos de adopción (los más críticos)

⚠️
El profesorado percibe el sistema como más trabajo, no menos
Mitigación: el MVP debe demostrar en la primera sesión que introduce los RA una vez y obtiene el 70% de su programación generada. Si el primer día no ve valor, no vuelve. La experiencia del docente en la Fase 2 es lo más importante.
⚠️
El jefe de departamento no lidera la implantación
Mitigación: el jefe es el usuario con más ganancia inmediata (visión del ciclo, alertas automáticas, menos emails). Hay que hacer la demo primero con él/ella, no con el docente.
⚡
Resistencia al cambio en centros con tradición de PDF en papel
Mitigación: el sistema siempre exporta a PDF. No elimina el documento final — lo genera. Eso reduce la percepción de riesgo ante inspección.

Decisiones clave a tomar antes de empezar

1️⃣
¿Piloto en un solo centro o multi-centro desde el inicio?
Recomendación: un solo centro (IES Fernando Sagaseta, Telde) para el MVP. Escalar después.
2️⃣
¿Desarrollo propio o buscar colaboración técnica?
Si no hay perfil técnico disponible: buscar desarrollador freelance Next.js o colaboración con universidad (TFG/TFM de ingeniería informática).
3️⃣
¿Financiación pública o desarrollo privado?
El proyecto encaja en convocatorias de innovación educativa del Gobierno de Canarias y en programas de digitalización de la FP del MEFP. Vale la pena explorar antes de financiación propia.
4️⃣
¿Licencia open source o producto cerrado?
Open source (MIT o similar) facilita la adopción en otros centros y puede atraer contribuidores. Modelo de negocio posible: hosting y soporte de implantación.
